Abstraction

By depending on abstract types (like interfaces) instead of concrete types, you can reduce the coupling between different parts of a program, making it more flexible and easier to change and maintain.

Program to an interface, not an implementation

aka "coding to interface"

Code should depend on interface abstractions and not on concrete classes. This can increase the flexibility of the code.

Different languages have different constructs for such abstractions:

  • trait ‚Äî Rust
  • interface, abstract class ‚Äî Java, Kotlin
  • protocol ‚Äî Swift
  • abc.ABC ‚Äî Python

Some languages bank on the use of duck typing to achieve this abstraction ‚Äî "if it looks like a duck, swims like a duck, and quacks like a duck, then it probably is a duck."

Decorator

Wrap an inner functionality with an outer functionality (the decorator).

Text, BoldFace text

Given an inner functionality Text:

class Text(text: String) {
	fun render() -> String {
		return text
	}
}

we wrap it with an outer functionality BoldFace:

class BoldFace(wrapped: Text) {
	fun render() -> String {
		return "<b>{wrapped.render()}</b>"
	}
}

Facade

Hide lower-level structures with a higher level structure.

(Tyre, Tank) vs. Car

Given low-level structures Tyre and Tank:

class Tyre(val pressure: Int)

class Tank(val fuel_level: Int)

we define a higher-level structure called Car:

class Car(
    private val tank: Tank,
    private val tyres: List<Tyre>,
) {
    fun tyre_pressures() -> List<Int> {
        return tyres.map { it.pressure }
    }

    fun fuel_level() -> Int {
        return tank.fuel_level
    }
}

Adapter

Changing an object to fit to an existing implementation.

UDP

Given that we have a method or an object (‚Äúadapter client‚Äù) expects another object to have a write method.

def log(message, destination):
	destination.write(f"{message}")

We want to write a message to a UDP socket using this.

But UDP objects don't have this method. 

import socket
udp_socket = socket.socket(...)
udp_socket.send("hello", ...)

So we write a class for this (‚Äúadapter‚Äù) that will have this send method.

class UdpSocketAdapter:

  def __init__(self, ip, port):
      self._socket = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
      self._ip = ip
      self._port = port

  def write(self, message):
      self._socket.send(message, (self._ip, self._port))

Such that now

udp_socket_adapter = UdpSocketAdapter()
log(message="hello", destination=udp_socket_adapter)

An HTTP request is passed through a series of filters ("filter chain") before rreaching the destination controller.

This is an example of the Chain of Responsibility pattern beacuase it allows the filterrs to be organised in a chain. Each filter has the ability to either handle the request or pass it along.

A common idiom is

Python

content_filter = ContentFilter([offensive_filter, ads_filter, porno_video_filter])
filtered_content = content_filter.filter(content)

class ContentFilter(object):
    def __init__(self, filters=None):
        self._filters = list()
        if filters is not None:
            self._filters += filters

    def filter(self, content):
        for filter in self._filters:
            content = filter(content)
        return content

Interface Segregation

Break down ("segregation") a large interface into smaller, specific interfaces.

Composition over inheritance

You don‚Äôt have to keep inheriting.

Instead of

class DbObject:
  pass

class User(DbObject):
  pass

We can do

class User:
  _methods = ['get', 'save', 'delete']

  def __init__(self, methods):
    self._methods = methods

  def __getattr__(self, attr):
    if attr in self._methods:
        return getattr(self._methods, attr)

  def get(self):
    pass

  def save(self):
    pass

  def delete(self):
    pass

Such that we have some users who can only get stuff

user1 = User("get")
user1.get(...)

And some users have only delete rights

user2 = User("delete")
user2.delete(...)

Immutability

Rust

Immutability is a fundamental part of Rust's language design.

By default, all declared variables in Rust are immutable.

#![allow(unused)]
fn main() {
let x = 5;
x = 6;  // compile-time error
}

JavaScript

Immutability does not apply to all objects.

While this will fail:

const x = 1;
x = 2;

this won't:

const person = {"name": "raimi"};
person["name"] = "adam";

Immutability and shadowing

While Rust allows name shadowing, the value that it holds is still immutable.

#![allow(unused)]
fn main() {
let x = 1;
let x = 2;
}
